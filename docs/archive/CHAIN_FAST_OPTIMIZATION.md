# Chain-Fast 优化思路说明

这份文档不记录代码改了什么，而是回答四个更实际的问题：

- `chain-fast` 现在到底慢在哪里
- `mod4` 和后来的 `mod` 预筛有什么区别
- 大数计算会不会很快变成主瓶颈
- 后面到底该优先做数据库、并行，还是继续减搜索空间

## 一、当前真正慢在哪里

先说结论：

- 当前主瓶颈还是**海量 pair 的前置过滤**
- 不是生成本原勾股数
- 不是数据库
- 也还不是最后那几次“大整数是否为平方”的精确判定

大白话就是：

- 现在最花时间的，不是“最后检查那一下”
- 而是“为了走到最后检查，要先扫掉太多候选”

以 `max_hyp=50000, backend=python, workers=1` 为例，已有 profile 大致是：

- `time_filter_s ≈ 48s`
- `time_c3_s ≈ 7s`
- 其余步骤都很小

这说明当前主要矛盾还是：

- 要做的 pair 太多
- 每个 pair 前面那套 `gcd / 整除 / 构造 A,B,N / 基础筛` 累计太贵

## 二、`mod4` 和 `mod1680` 有什么区别

它们看起来都叫“mod 筛”，但工程上不是一回事。

### 1. 当前默认就有的 `mod4`

默认路径里本来就有很便宜的必要条件筛，主要是：

- 奇偶筛
- `mod 4` 筛

这些筛的特点是：

- 逻辑短
- 不需要额外查大表
- 本来就和主过滤逻辑贴在一起

可以把它理解成：

- “路过顺手检查一下”

### 2. 后来实验的 `mod1680`

后来的实验筛固定用了：

- `16`
- `3`
- `5`
- `7`

把它们合起来以后，最小公倍数是：

- `1680`

所以实验筛做的是：

- 先算 `A mod 1680`
- 再算 `N mod 1680`
- 然后查一张表，判断 `A² + N²` 在这些模条件下还有没有可能是平方

它的特点是：

- 筛得更强
- 但每个候选都多了一层 `%` 和查表成本

所以两者的区别不是“只换了一个数字”，而是：

- `mod4` 是很轻的结构筛
- `mod1680` 是额外加上去的一层通用筛

## 三、为什么 `mod1680` 没有变快

实验结果很重要：

- 它确实把进入精确 `C3` 的候选砍掉了很多
- 但总 wall time 没有变快，反而略慢

原因不复杂：

- 省下来的 `isqrt(C3)` 时间
- 还不够支付新筛自己带来的 `%` 和查表成本

也就是说，这版实验说明的是：

- “筛得更多”不等于“跑得更快”

当前这版 `mod` 预筛属于：

- 数学上成立
- 工程上暂时不划算

所以它适合保留成实验开关，不适合直接变成默认优化。

## 四、大数会不会变成新瓶颈

会越来越贵，但**当前还不是第一瓶颈**。

这里要分清两件事：

### 1. 数学量级确实会变大

当 `max_hyp = H` 时：

- 腿长大约是 `O(H)`
- `A`、`N` 这类量大约是 `O(H²)`
- `A² + N²` 这类完全平方检查大约是 `O(H⁴)`

所以从数学上讲，后面遇到的大整数一定会越来越大。

### 2. 但当前真正先把程序拖慢的，还是 pair 数量

比如 `max_hyp=100000` 时，triple 数大约三万多，但 pair 已经到十亿级。

所以目前更像是：

- 单次大数运算会变贵
- 但真正先爆炸的，还是“要做多少次”

一句话说：

- 现在的主问题还是 `n²`
- 还不是“大整数单次成本已经压过一切”

## 五、数据库解决什么，不解决什么

数据库和 `resume` 很有必要，但它解决的是**长跑可靠性**，不是算法速度。

数据库能解决：

- 跑一半停了不用重来
- 不同参数实验能长期积累
- near-miss、结果、profile 可以留档
- 以后做分块长跑更方便

数据库不能直接解决：

- `chain-fast` 的主循环为什么慢
- 为什么搜索空间这么大
- 哪些 pair 根本不该算

所以数据库是“工程底座”，不是“主性能优化点”。

## 六、下一步最值得做什么

当前最合理的优先级大致是：

### 1. 优先继续减少搜索空间

这是最值钱的方向。

重点不应该是继续加“很晚才发生的筛”，而是想办法更早排除整批 pair。

最值得关注的方向包括：

- 更早的结构筛
- 更早的同余类分桶
- 从 `t1r / s2r / gcd(t1,s2)` 结构里挖更便宜的必要条件

### 2. 继续保留数据库和断点续跑

如果后面要跑很大范围，这是必须有的工程保障。

### 3. 并行仍然有价值

并行不能改变复杂度，但对大范围长跑仍然很有帮助。

### 4. 不要急着继续堆“后置 `mod` 筛”

这次实验已经说明：

- 在当前插入位置上，筛得更多不代表更快

如果后面还要试 `mod`，更值得尝试的是：

- 让它发生得更早
- 或者让它和分桶/分组结合

而不是简单再换更大的模数。

## 七、当前一句话判断

如果只用一句话概括现在的状态，那就是：

- `chain-fast` 目前最缺的不是“再多一个晚筛”
- 而是“更早、更便宜地砍掉整批无望候选”

后面如果真有明显提速，更可能来自：

- 更强的结构剪枝
- 更早的分桶
- 或者更适合长跑的并行与持久化协作

而不是单纯继续堆一个更重的 `mod` 检查。
