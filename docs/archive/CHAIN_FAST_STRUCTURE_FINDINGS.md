# Chain-Fast 结构统计首轮结论

这份文档只记一件事：

- `chain-fast --bucket-stats` 跑到较大范围以后，**目前已经能比较有把握地看出什么**

它不是接口说明，那部分已经写在：

- [docs/CHAIN_FAST_BUCKET_STATS.md](./CHAIN_FAST_BUCKET_STATS.md)

这里更关心的是实测结论。

## 一、这轮看了什么

本轮主要看三类结构桶：

- `g_bucket`：`g = gcd(t1, s2)`
- `delta_bucket`：`s1-t1`、`s2-t2` 的符号 / 大小区间 / `mod 8`
- `residue_bucket`：`s1,t1,s2,t2 (mod 8)` 组合

并且重点不是看“出现了多少”，而是看：

- 过完基础筛以后，还剩多少
- 这些剩下的东西里，谁更容易活到 `C3`
- 哪些桶更容易出现“很接近解”的 `near-miss`

## 二、这轮用了哪些实测

### 1. `max_hyp=5000`

- triples: `1584`
- pairs: `2,509,056`
- after_filters: `620,292`
- `C3` pass: `37`
- `C4` pass: `0`
- wall time: `4.35s`

### 2. `max_hyp=10000`

- triples: `3186`
- pairs: `10,150,596`
- after_filters: `2,510,542`
- `C3` pass: `72`
- `C4` pass: `0`
- wall time: `13.19s`

### 3. `max_hyp=100000`

这里分成两件事：

- 带 `--near-miss` 的 run，搜索本身完成了，但最后写 SQLite 时因为整数过大报错
- 去掉原始 near-miss 明细持久化，只保留 `profile + bucket stats` 后，run 完整成功

完整成功那次的结果是：

- triples: `31,838`
- pairs: `1,013,658,244`
- after_filters: `250,042,288`
- `C3` pass: `318`
- `C4` pass: `0`
- wall time: `1192.08s`

## 三、最清楚的结论

### 1. `C3` 仍然极稀少，主问题还是“前面算太多”

`max_hyp=100000` 时：

- 基础筛后还剩 `250,042,288`
- 真正过 `C3` 的只有 `318`

换成更直白的话：

- 基础筛后大约一百万个候选里，只有一两个能活到 `C3`

这说明一件很重要的事：

- 现在最值钱的优化，还是更早减少搜索空间
- 不是继续优化最后那一下 `isqrt`

### 2. `g_bucket` 的信号已经比较稳定

在 `100000` 这轮里，`g` 越大的桶，`C3` 命中率明显越高。

最显眼的几档是：

- `513+`
- `257-512`
- `129-256`
- `65-128`

反过来看：

- `g=3/4/5` 这类小桶虽然样本很多
- 但它们的命中率明显不如上面这些大 `g` 桶

这还不等于“以后可以直接把小 `g` 全砍掉”，但已经很像真结构，不像随机噪声了。

### 3. `residue_bucket` 也开始出现稳定模式

目前最容易活到 `C3` 的 residue 结构，反复集中在这类模式：

- `t1 ≡ 0 (mod 8)`
- `s2 ≡ 0 (mod 8)`
- `s1` 和 `t2` 往往落在 `1` 或 `7 (mod 8)`

这件事很重要，因为它说明：

- `mod 8` 结构不只是“出现得多”
- 它很可能真的和命中率有关

后面如果要继续做结构筛，这一块很值得优先深挖。

### 4. `delta_bucket` 目前有信号，但还不够干净

现在能看到的趋势是：

- 很多前排桶都带一个或两个 `128+` 的大差值区间

这说明：

- `|s-t|` 很大的结构，可能更容易活到 `C3`

但问题是：

- 这条线现在还不够整齐
- 还看不出一个特别简单、特别硬的必要条件

所以它更像“继续观察的方向”，还不像“马上写成剪枝”的方向。

### 5. 最接近解的老模式还在

到 `100000` 为止，最接近解的那些 `near-miss` 结构，还是反复落在你之前就见过的模式上，比如：

- `(7, 24, 45, 28)`
- `(28, 45, 24, 7)`
- `(231, 520, 765, 476)`
- `(476, 765, 520, 231)`

这说明：

- 大范围以后，并没有突然冒出一大类全新的“更像真解”的结构
- 旧的那些近解家族，依然是现在最值得盯的对象

## 四、这轮顺便暴露出的工程边界

### 1. 原始 near-miss 明细已经碰到 SQLite `INTEGER` 上限

`max_hyp=100000` 时，带 `--near-miss` 的 run 在最后写库时报错：

- `OverflowError: Python int too large to convert to SQLite INTEGER`

这说明现在这条路有明确边界：

- 小中范围还能存
- 到 `10w` 级别以后，原始大整数明细已经不能再放心按 SQLite `INTEGER` 硬存

所以如果以后还要继续往大跑，更稳的方向会是：

- 存聚合统计
- 存排序后的少量代表样本
- 或者把超大整数改成别的持久化表示

### 2. 当前 `--bucket-stats` 更像分析开关，不像长跑默认开关

这轮 `100000` 的 `bucket-stats` run 用了大约：

- `1192s`，也就是接近 `20` 分钟

而此前不带这套结构聚合的 `100000, backend=python, workers=8` 基线记录，大约是：

- `71.8s`

这说明当前这版结构桶统计虽然有用，但代价也很大：

- 它已经足够适合“做一次证据采集”
- 但还不适合作为默认长期挂着的生产开关

一句话说：

- 现在的 `--bucket-stats` 是分析工具，不是免费午餐

## 五、现在最值得继续看的两条线

如果只按这轮数据排优先级，我会先看这两条：

### 1. `g_bucket`

原因很简单：

- 信号最稳定
- 样本量也够大
- 不是靠几个偶然小桶撑起来的

### 2. `residue_bucket`

它的好处是：

- 规则更轻
- 更有希望往“便宜前筛”改写

相比之下，`delta_bucket` 现在还更像需要继续积累证据，而不是立刻动刀。

## 六、当前一句话判断

如果现在只留一句话，那就是：

- `10w` 结构统计已经足够说明，后面该优先做的是更早的结构筛，尤其是围绕 `g` 和 `mod 8` 的前段筛

同时也要记住另一半现实：

- 当前这版 `--bucket-stats` 本身太重，而且原始 near-miss 明细已经撞上 SQLite 的整数上限

所以后面真正要落地时，应该是：

- 用这轮结果指导新的轻量筛
- 而不是把这套重统计一直挂在所有长跑任务上
