# Chain-Fast `mod` 筛实验记录

这份文档只回答两个问题：

- `mod` 筛到底在做什么
- 它在当前实现里到底有没有加速

结论先说：

- 这版 `mod` 筛在数学上是有效的
- 它确实能大幅减少进入精确 `C3` 检查的候选数
- 但在当前 Python 实现里，总 wall time **没有更快，反而略慢**

所以它现在是一个**实验开关**，不是默认优化。

## 一、它在筛什么

`chain-fast` 里最贵的一步之一，是对海量候选检查：

- `C3: A² + N²` 是否是完全平方数

`mod` 筛的想法很简单：

- 如果一个数在模 `m` 意义下都不可能是平方数
- 那它在整数里也不可能是平方数

比如平方数模 4 只可能余 `0` 或 `1`。  
如果某个候选算出来 `A² + N² ≡ 2 (mod 4)`，那它就可以直接丢掉，连 `isqrt` 都不用做。

这版实验固定用了四个小模数：

- `16`
- `3`
- `5`
- `7`

实现上会先查一张预计算表，判断 `A² + N²` 在这些模数下是否仍“有可能”是平方。  
只有通过这层筛的候选，才会进入真正的 `isqrt(C3)`。

## 二、实测结果

命令都用：

```bash
uv run python scripts/search.py chain-fast ... --backend python --profile --no-progress
uv run python scripts/search.py chain-fast ... --backend python --profile --no-progress --mod-sieve
```

### 2.1 `max_hyp=5000`, `workers=1`

| 模式 | wall time | after_filters | after_c3_mod | c3_pass |
|------|-----------|---------------|--------------|---------|
| baseline | 0.8s | 620,292 | 620,292 | 37 |
| `--mod-sieve` | 0.8s | 620,292 | 213,509 | 37 |

观察：

- 进入精确 `C3` 的候选少了约 `65.6%`
- 但总时间几乎没变

### 2.2 `max_hyp=50000`, `workers=1`

| 模式 | wall time | after_filters | after_c3_mod | c3_pass | `time_mod_sieve_c3_s` | `time_c3_s` |
|------|-----------|---------------|--------------|---------|------------------------|-------------|
| baseline | 76.3s | 62,546,538 | 62,546,538 | 202 | 0.000s | 7.060s |
| `--mod-sieve` | 82.8s | 62,546,538 | 21,657,375 | 202 | 11.397s | 2.579s |

观察：

- 精确 `C3` 候选少了约 `65.4%`
- `C3` 精确检查时间从 `7.060s` 降到 `2.579s`
- 但新增的 `mod` 筛本身花了 `11.397s`
- 所以总时间反而慢了约 `8.5%`

### 2.3 `max_hyp=100000`, `workers=8`

| 模式 | wall time | after_filters | after_c3_mod | c3_pass | `time_mod_sieve_c3_s` | `time_c3_s` |
|------|-----------|---------------|--------------|---------|------------------------|-------------|
| baseline | 115.4s | 250,042,288 | 250,042,288 | 318 | 0.000s | 80.370s |
| `--mod-sieve` | 119.8s | 250,042,288 | 86,768,373 | 318 | 129.275s | 27.752s |

观察：

- 精确 `C3` 候选少了约 `65.3%`
- `C3` 精确检查时间明显下降
- 但 `mod` 筛本身仍然更贵
- 所以总 wall time 还是更慢

## 三、这说明什么

这次实验说明了一件很重要的事：

- 当前瓶颈不只是“`isqrt(C3)` 太多”
- 还包括“在 Python 里额外做一遍取模和查表，也很贵”

换句话说：

- 这版筛法在“过滤率”上是成功的
- 但在“总运行时间”上是失败的

所以当前最合理的判断是：

- `--mod-sieve` 保留为实验开关
- 不作为默认优化

## 四、下一步如果还想继续做 `mod`

如果后面还要沿这条路继续试，优先级应该是：

- 让 `mod` 判断本身更便宜，而不是只追求筛得更多
- 尝试更早的分桶/分组，减少真正进入配对主循环的 pair
- 优先找能在 `gcd` 和基础筛之前就排掉整批组合的结构条件

简单说，这次结果不是“`mod` 思路没用”，而是“这版 `mod` 插入点和实现成本，还不够划算”。
