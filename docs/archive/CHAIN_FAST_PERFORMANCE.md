# Chain-Fast 性能记录

这份文档记录 `chain-fast` 在加入 `--profile` 之后的首轮实测结果，目的是回答三个问题：

- 时间到底花在哪
- 数据库和 triple 缓存是不是当前瓶颈
- 下一步更值得做并行，还是更值得做剪枝

## 一、测试方式

本轮测试都使用：

```bash
uv run python scripts/search.py chain-fast ... --profile --no-progress
```

而且先不写数据库，所以 `time_db_write_s = 0`，测到的几乎都是纯计算时间。

测试机当前看到的逻辑核数是 `10`。

## 二、核心结论

先说结论：

- 当前主瓶颈不是生成本原勾股数
- 当前主瓶颈也不是数据库
- 当前主瓶颈是海量三元组对的前置过滤
- `python` 路径用多进程很有价值
- `numpy` 在安全范围内非常强，但安全上限仍然在 `max_hyp ≈ 36000`

换句话说，后续优化优先级大致是：

1. 更强的过滤 / 剪枝
2. `python` 路径多进程
3. triple 缓存与数据库复用

### 3.2 多进程扩展性（python, max_hyp=50000）

| workers | wall time | speedup vs 1 worker |
|---------|-----------|---------------------|
| 1 | 75.7s | 1.0× |
| 2 | 53.9s | 1.4× |
| 4 | 35.2s | 2.2× |
| 8 | 22.7s | 3.3× |
| 10 | 24.3s | 3.1× |

观察：

- 这台 10 核机器上，`8 workers` 左右已经接近甜点
- `10 workers` 反而略慢，说明调度和合并开销开始冒头

### 3.3 更大范围趋势

`max_hyp=100000, backend=python, workers=8`

- wall time: `71.797s`
- triples: `31,838`
- pairs: `1,013,658,244`
- after_filters: `250,042,288`
- c3_pass: `318`
- c4_pass: `0`

这说明即使 pair 数已经到 10 亿级别，只要用并行，仍然能把 wall time 压在一分钟多一点。但也说明搜索空间极大，而真正走到后面的候选极少。

### 3.4 安全范围内的 numpy + 并行

`max_hyp=30000, backend=numpy, workers=8`

- wall time: `2.731s`
- triples: `9,550`
- pairs: `91,202,500`
- after_filters: `22,501,068`
- c3_pass: `141`
- c4_pass: `0`

这说明在 `numpy` 安全范围内，`numpy + 多进程` 的实际性能非常强。

## 四、profile 怎么解读

以 `max_hyp=50000, backend=python, workers=1` 为例：

- `time_generate_triples_s = 0.003`
- `time_filter_s = 48.323`
- `time_c3_s = 7.191`
- `time_c4_s = 0.000`
- `time_dedup_s = 0.001`
- `time_db_write_s = 0.000`

这里最重要的是：

- triple 生成几乎可以忽略
- 去重几乎可以忽略
- 数据库没开时自然也是 0
- 绝大多数时间都花在 filter

所以“把本原勾股数存起来能不能显著省时间”这个问题，当前答案是：

- 能省一点点准备时间
- 但省不到主瓶颈

triple 缓存更像是为了：

- 复用同一批输入
- 做后续统计分析
- 为 future 分析打底

而不是为了显著降低 wall time。

## 五、当前最合理的下一步

从这轮数据看，后续最值得继续做的事情是：

- 优先研究更强的模筛或结构筛，让更多 pair 在进入 `C3` 前就被砍掉
- 在 `python` 路径上继续利用多进程，默认值可优先考虑 `workers≈8`
- 把 triple 缓存保留为“分析底座”，但不要把它当成主性能优化点

简单说，当前项目已经有了比较清楚的性能画像。后面如果要继续提速，应该优先少算，而不只是优先少生成。
