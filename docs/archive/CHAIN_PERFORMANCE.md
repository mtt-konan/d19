# Chain 性能记录

这份文档只记一件事：

- 默认 `chain` 路线现在花时间的地方到底在哪

这里不讨论 `--require-square`。
你现在想走的是默认 `chain` 路线，也就是先找长方形 4-cycle，再看后面还能怎么剪。

## 一、测试口径

本轮只测默认 `chain`：

```bash
uv run python scripts/search.py chain --max-val <N> --no-progress --top 0
```

为了看清瓶颈，我把总时间拆成两段：

- `build_adjacency`
  先建勾股邻接表。对每个 `a`，扫一遍 `1..max_val` 的 `b`，保留满足 `a^2+b^2` 是平方的配对。
- `search_cycles`
  在邻接表里找 4-cycle，也就是按 `a -> b -> c -> d` 去闭环。

所有数字都来自同一台机器上的本地实测。
它们适合做同机前后对照，不适合拿去当跨机器绝对结论。

|----------:|------------------:|----------------:|--------:|------------:|-------------:|---------------------:|----------:|
| `5,000`   | `0.112s` | `0.065s` | `0.177s` | `15,844`  | `231,568`   | `484,296`   | `1,005` |
| `10,000`  | `0.391s` | `0.180s` | `0.571s` | `33,198`  | `518,896`   | `1,081,790` | `2,843` |
| `20,000`  | `1.486s` | `0.694s` | `2.180s` | `62,854`  | `901,906`   | `1,878,782` | `7,456` |
| `50,000`  | `8.033s` | `2.716s` | `10.749s` | `173,500` | `3,060,748` | `6,394,036` | `25,075` |
| `100,000` | `35.129s` | `7.487s` | `42.616s` | `371,728` | `7,611,552` | `15,901,816` | `59,600` |
| `150,000` | `79.937s` | `13.134s` | `93.072s` | `579,296` | `12,907,026` | `26,951,548` | `97,400` |

这些列怎么读：

- `adj_edges`
  邻接表里一共保留了多少条勾股边。
- `abc_checks`
  程序一共走了多少次 `a -> b -> c` 组合。
- `d_candidates_total`
  这些 `abc` 组合一共生成了多少个候选 `d`。
- `results`
  去重后的默认 `chain` 结果数。

## 三、先看结论

- 当前 `chain` 最慢的地方是 `build_adjacency`，不是最后的结果整理。
- 范围一大，前半段会比后半段更快变成主瓶颈。
- `search_cycles` 也在涨，但它现在还是第二瓶颈。
- 邻接表很适合落盘复用。`max_val=150000` 时，`a,b,h` 三列落成 SQLite 实测大约 `19.7 MB`。
- 这批邻接表数据不会碰到 SQLite `int64` 上限。`150000` 这一档里最大的斜边只有 `211775`。

拿几档大一点的值看最直观：

- `max_val=50,000` 时，建邻接表花 `8.0s`，搜环花 `2.7s`
- `max_val=100,000` 时，建邻接表花 `35.1s`，搜环花 `7.5s`
- `max_val=150,000` 时，建邻接表花 `79.9s`，搜环花 `13.1s`

所以你后面如果要做优化，第一反应不该是“怎么更快打印结果”，而该先盯：

- 邻接表怎么更便宜地建
- 进入 `d_set` 之前，怎么少走很多 `a -> b -> c`

## 四、落盘和最大样例

`max_val=150000` 这档，我额外做了两件实测。

### 1. 邻接表落盘大小

把邻接表存成一张 SQLite 表，行级只存：

- `a`
- `b`
- `h`

一共是：

- `579,296` 行
- 文件大小约 `20,680,704` 字节
- 也就是大约 `19.7 MB`

这说明默认 `chain` 的邻接表很适合做本地 SQLite 缓存。

### 2. 最大边值那组结果

按“最大边值”排，`max_val=150000` 这一档里最大的 canonical 默认 `chain` 结果是：

- `(33750, 81000, 73125, 150000)`

它对应的外接矩形是：

- `106875 × 231000`

四条边的分解是：

- `33750² + 81000² = 87750²`
- `81000² + 73125² = 109125²`
- `73125² + 150000² = 166875²`
- `150000² + 33750² = 153750²`

这组结果不是正方形解。它只是当前默认 `chain` 结果里，最大边值顶到 `150000` 的那一组。

## 五、代码里到底慢在哪

### 1. 建邻接表

代码在 [src/rational_distance/search_chain.py](../src/rational_distance/search_chain.py) 的 `_build_adjacency()`。

这段会做一件很重的事：

- 对每个 `a`
- 扫一整排 `b = 1..max_val`
- 算 `a^2 + b^2`
- 开方
- 判断是不是完全平方

你可以把它看成在扫一张很大的 `a x b` 表。

这就是为什么 `max_val` 从 `50,000` 长到 `100,000` 时，`build_adjacency` 会从 `8.0s` 涨到 `35.1s`。
这段本身就带着很强的平方级味道。

### 2. 搜 4-cycle

代码在同一个文件里的 `find_chains()` 主循环。

这段在做：

- 枚举 `a`
- 枚举 `adj[a]` 里的 `b`
- 枚举 `adj[b]` 里的 `c`
- 再算 `adj[c] ∩ adj[a]` 得到候选 `d`

它慢在几件小事反复堆起来：

- `a -> b -> c` 的嵌套遍历很多
- 集合交集很多
- `sorted(d_set)` 很多
- 对称去重 `min(_symmetry_group(...))` 也会重复做

这些小成本单次不重，次数一大就会堆出明显时间。

## 六、这份数据对优化有什么用

这份表最适合做基线。

你后面不管做哪种优化，都该先问两个问题：

1. 你省的是 `build_adjacency`，还是 `search_cycles`？
2. 你省的是常数，还是把 `abc_checks` / `d_candidates_total` 真砍掉了？

如果一个改动只让最后输出变短，或者只让结果对象少建一点，它通常不会动到主瓶颈。

如果一个改动能做到下面任意一条，它才更像真优化：

- 少扫很多 `a,b` 配对
- 少让很多 `a -> b -> c` 走进循环
- 少做很多集合交集
- 复用已经建好的邻接表

## 七、一句话判断

当前默认 `chain` 的主要时间花在两段：

- 先把整张勾股邻接表扫出来
- 再在这张图里找四边闭环

其中第一段更慢，而且随着范围变大，第一段会越来越像主瓶颈。

## 八、异号筛前移之后，到底快了多少

这轮我把 `--diagonal-sign-sieve` 从“结果出来后再筛”改成了“搜索过程中就剪掉”，
而且结果缓存也单独拆成了两种模式：

- `default`
- `diagonal_sign`

邻接表缓存还是共用一套。
如果你用旧版 `chain` 缓存库，现在会直接提示重建，因为 SQLite schema 已经升到 `v2`。

### 1. `150000` fresh run 对比

测试命令是：

```bash
uv run python scripts/search.py chain --max-val 150000 --db .cache/bench-chain-default-150k-v2.sqlite3 --no-progress --top 1
uv run python scripts/search.py chain --max-val 150000 --db .cache/bench-chain-diag-150k-v2.sqlite3 --diagonal-sign-sieve --no-progress --top 1
```

对应的 SQLite run 记录如下：

| 模式 | `build_adjacency_s` | `search_cycles_s` | `query_results_s` | `total` | `new_results` | `filtered` |
|------|--------------------:|------------------:|------------------:|--------:|--------------:|-----------:|
| `default` | `80.007s` | `13.249s` | `0.277s` | `93.912s` | `97,400` | `0` |
| `diagonal_sign` | `79.904s` | `12.427s` | `0.090s` | `92.786s` | `32,191` | `65,209` |

这组数字怎么理解：

- 异号筛前移后，`search_cycles` 确实变快了。
- 但它只从 `13.249s` 降到 `12.427s`，只省了大约 `0.822s`。
- 总时间只从 `93.912s` 降到 `92.786s`，只省了大约 `1.126s`。
- 主因还是没变：最重的仍然是前面的邻接表构建，依旧在 `80s` 左右。

所以这次改动的结论很明确：

- 它是“真前筛”，不是假过滤。
- 它确实让后半段搜索更轻。
- 但在 `150000` 这一档，总耗时只小幅下降，因为前半段太重了。

### 2. 它到底筛掉了多少

`150000` 这一档里：

- 默认 `chain` 结果数是 `97,400`
- 前移后的异号筛模式保留 `32,191`
- 一共筛掉 `65,209`

也就是说，按 canonical 结果数看，它大约筛掉了三分之二。

### 3. 纯查表路径是否正常

同样的命令再跑一遍，不重建缓存时，SQLite run 记录是：

| 模式 | `total` | `query_results_s` | `new_adjacency_edges` | `new_results` |
|------|--------:|------------------:|----------------------:|--------------:|
| `default` | `0.353s` | `0.351s` | `0` | `0` |
| `diagonal_sign` | `0.076s` | `0.074s` | `0` | `0` |

这说明现在两件事都已经成立：

- 邻接表确实只建一次
- 结果也确实按模式持久化了，后续直接查表即可

而且因为 `diagonal_sign` 结果表更小，纯查表时也会更快一些。
