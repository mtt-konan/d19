# `concordant` 安全前筛说明

这份文档只讲一件事：

- `concordant` 这条线里，哪些前筛现在已经可以放心落地
- 哪些想法看起来诱人，但其实太弱，筛不掉任何东西

尽量用普通话说，不堆太多术语。

## 1. 先说结论

当前 `concordant --max-hyp` 批量入口上，第一版真正能落地的安全前筛是：

- `A` 必须是奇数
- `B` 必须是奇数
- `(A + B) % 4 == 0`

注意这里说的是：

- 当前 `generate_ab_pairs()` 产出的 **reduced `(A,B)`**
- 这是 **完整 4-chain** 的必要条件
- 不是充分条件

大白话就是：

- 不满足这三个条件的 pair，可以提前排掉
- 满足这三个条件的 pair，也只是“还没死”，不代表真的有解

## 2. 为什么之前那版 `mod1680` 是空筛

之前试过一版 pair 级的局部可解性筛。

它问的是：

- 给定一对 `(A,B)`
- 在模 `16,3,5,7` 这些小模数下
- 是否存在某个 `N` 的剩余类
- 能让 `N^2 + A^2` 和 `N^2 + B^2` 都看起来“有可能是平方”

这件事看上去合理，但问题是它太弱了。

因为你永远可以取：

- `N ≡ 0 (mod m)`

这时：

- `N^2 + A^2 ≡ A^2 (mod m)`
- `N^2 + B^2 ≡ B^2 (mod m)`

而 `A^2`、`B^2` 本来就是平方剩余。

所以结果就变成：

- 对每个模数 `m`，任何 `(A,B)` 都能过
- 合起来的 `mod1680` 查表全是 `True`
- 这不是实现写错了，而是数学条件本身没有筛力

这和 `chain-fast` 那边的 `mod1680` 不是一回事。

区别是：

- `chain-fast` 筛的是某个已经构造出来的表达式，比如 `A^2 + N^2`
- `concordant` 那版筛的是“这对 `(A,B)` 是否存在某个 `N mod m`”

后一种因为 `N ≡ 0` 这个万能见证，会直接塌成空筛。

## 3. 为什么把完整四条拉回来以后，会出现更强的 2-adic 条件

`concordant` 单独看的是后两条：

- `N^2 + A^2 = square`
- `N^2 + B^2 = square`

但完整 4-chain 还要把前两条一起拉回来。设：

- `b = A + B - N`

那么完整检查其实是四条：

- `N^2 + A^2 = square`
- `N^2 + B^2 = square`
- `b^2 + A^2 = square`
- `b^2 + B^2 = square`

也就是说，`N` 和 `b` 要分别和 `A`、`B` 都能拼成勾股型关系。

一旦把这四条一起看，2 的幂次数，也就是常说的 2-adic 信息，就不再是“可有可无”的了。

## 4. 这版安全筛背后的直观推导

这里只写当前工程里真正要用的那一层。

### 4.1 先看奇偶

对于一个平方和：

- 两个奇数平方相加，结果 `≡ 2 (mod 4)`
- 这不可能是完全平方

所以在一条勾股条件里：

- 两边不能同时是奇数

回到完整 4-chain，`A` 和 `B` 是对边，`N` 和 `b` 是另外一对对边。  
如果要让四条都成立，当前 reduced `(A,B)` 批量入口里，最后能活下来的情况必须收缩到：

- `A`、`B` 都是奇数

混合奇偶的 pair 在完整链里会卡死，不能作为这条主线里的完整候选继续送进 PARI。

### 4.2 再看 `mod 4`

当 `A`、`B` 都是奇数时：

- 要让 `N^2 + A^2` 成为平方，`N` 必须是偶数
- 要让 `b^2 + A^2` 成为平方，`b` 也必须是偶数

再继续往下看一个台阶：

- 对奇数 `A`，如果 `N ≡ 2 (mod 4)`，那么 `N^2 + A^2 ≡ 4 + 1 ≡ 5 (mod 8)`
- 但 `5` 不是平方剩余模 `8`

所以：

- `N` 不能只是偶数，必须是 `4` 的倍数
- 同理，`b` 也必须是 `4` 的倍数

而我们又有：

- `N + b = A + B`

所以立刻得到：

- `A + B` 必须是 `4` 的倍数

这就是当前工程里能直接落成代码的必要条件：

- `A` odd
- `B` odd
- `(A + B) % 4 == 0`

## 5. 为什么这版只敢作用在 reduced `(A,B)`

这里要特别收口，不把结论说大。

这版实现只服务于：

- `generate_ab_pairs()` 产出的 reduced `(A,B)`

原因很简单：

- 这些 pair 已经做过约分和规范化
- 代码里批量 `concordant` 入口就是在扫这批 reduced pair
- 目前证明和工程判断都只对这条输入流负责

所以这不是一个“任意 `(A,B)` 都通用”的大定理接口。

它只是当前批量入口上，一个已经能放心用的必要条件筛。

## 6. 这版筛到底能做什么

它做的是：

- 在 `ellratpoints` 之前
- 提前排掉一大批从完整 4-chain 角度已经没戏的 reduced `(A,B)`

它不做的是：

- 不证明通过筛的 pair 就一定有 concordant `N`
- 不证明通过筛的 pair 就一定更接近完整解
- 不替代后面的椭圆曲线分析

所以它的定位很明确：

- 这是一个安全的“先别浪费 PARI 时间”的前筛

## 7. 小例子

- `(22, 35)`：一奇一偶，直接排掉
- `(5, 9)`：都是奇数，但 `5 + 9 = 14 ≡ 2 (mod 4)`，排掉
- `(7, 45)`：都是奇数，而且 `7 + 45 = 52 ≡ 0 (mod 4)`，这只是“允许继续分析”，不是说它真有完整链

## 8. 实测结果

这版安全前筛已经作为实验开关接进批量 `concordant --max-hyp`。

目前两组直接对照结果是：

| `max_hyp` | 原始 pair 数 | 送进 PARI 的 pair 数 | 拒绝比例 | `time_find_concordant_s` | wall time |
|---|---:|---:|---:|---:|---:|
| 1000 baseline | 24197 | 24197 | 0% | 23.777s | 23.9s |
| 1000 `--safe-pair-sieve` | 24197 | 2120 | 91.2% | 1.995s | 2.0s |
| 2000 baseline | 99311 | 99311 | 0% | 96.432s | 96.9s |
| 2000 `--safe-pair-sieve` | 99311 | 8220 | 91.7% | 7.671s | 7.8s |

该结果说明两件事：

- 这条 2-adic 必要条件不只是“数学上安全”，而且工程上确实有筛力
- 当前 `concordant` 的主耗时本来就集中在 `ellratpoints`，所以只要大幅减少送进 PARI 的 pair，wall time 也会跟着明显下降

要注意的是：

- 开了这个实验开关以后，输出不再是“全量 concordant 视图”
- 它会主动丢掉那些从完整 4-chain 角度已经不值得继续看的 pair

所以：

- 默认模式继续保留全量 concordant 观察
- `--safe-pair-sieve` 则是“完整链导向”的更快批量模式

详细记录见：

- [docs/work-logs/027-concordant-safe-pair-sieve.md](./work-logs/027-concordant-safe-pair-sieve.md)

## 9. 当前工程决定

所以当前 `concordant` 方向的工程口径固定为：

- 旧的 pair 级 `mod1680` 只保留为失败实验文档，不再保留为运行时功能
- 新的安全前筛用 `--safe-pair-sieve` 作为实验开关
- 默认批量模式仍然保留原来的“看所有 concordant pair”的语义
- 只有显式打开实验开关时，才切换到“完整链导向”的前筛分析
